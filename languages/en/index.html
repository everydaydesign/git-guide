Git guide for group work
Collaborate without overwriting each other

How Git works

Git saves snapshots of your code, so several people can work in the same project without overwriting each other. Here is the whole picture, from your first commit to a merged Pull Request.

01
The basics

A time machine for your code

Instead of saving copies like v1, v2_final, FINAL Git keeps track of every change for you.
You can always go back to an earlier version — and see who changed what, and when.

02
The big picture

How code flows in a group

One person creates the repo and invites the others.
Everyone pulls (pull) the latest version, each works on their own branch, and sends their work back via a Pull Request that is reviewed before it is merged.

Which template? The group repo is created from a template — website for HTML/CSS, or web-app for PHP/MySQL.

Golden rule: never work directly in main — always create your own branch.

03
Fundamentals

Stage and commit — like packing a bag

First, choose what to include (stage). Then close the bag and put a label on it (commit). Two steps, for a reason.

Step 1 · Stage
Choose what to include
Like choosing clothes for the bag — you pick which changed files belong together in this particular save.
login.phpstaged
style.cssstaged
test.phpnot selected
Step 2 · Commit
Close the bag, add a label
You save a snapshot of the selected files — with a message describing what you did.
COMMIT MESSAGE
Added login page

Commit

saves 2 files locally

Why two steps? You might have changed five files but only want to save the two that belong together right now. With stage you choose exactly what to include — so that every commit is a logical, understandable change.

05
Branches

Your own workspace, separate from main

A branch is a copy of the project where you can work freely without affecting others' code. When you are done, your branch is merged with main via a Pull Request.

Golden rule: never work directly in main. Always create a branch first.

Good branch names
  • contact
  • products
  • login
  • startpage
06
Sync

Pull first, push after

Pull pulls others' changes from GitHub to you. Push pushes your commits up. You always sync in both directions.

07
Review

No code enters main unreviewed

A Pull Request lets the group read your code before it is merged. Another person reviews — the only rule is that you don't review your own code.

The reviewer's decision
  • Approve
    Everything looks good → the PR can be merged into main.
  • Comment
    Questions or suggestions → discuss, no blocking.
  • Request changes
    Something needs fixing → the author changes and pushes again.
08
When it clashes

When two people change the same line

A conflict happens when two people have changed exactly the same line. Git doesn't know which version should win — so you have to choose. Usually the answer is to combine both.

Conflict in header.php
<<<<<<< HEAD (Erik)
<a href="products.php">Products</a>
=======
<a href="contact.php">Contact</a>
>>>>>>> login (Lisa)
Solution · keep both
<nav>
  <a href="products.php">Products</a>
  <a href="contact.php">Contact</a>
</nav>
Remove <<<, === and >>>, save, and commit.

Avoid conflicts: talk within the group about who works on which file — and pull often.

Quick checklist

Do once · project start
  • One person creates the repo from a template — website (HTML/CSS) or web-app (PHP/MySQL)
  • Invite the group as collaborators
  • Everyone accepts and creates their Codespace
  • Decide who does what
Do every work session
  • Switch to main + Pull
  • Create / switch to your branch
  • Stage → Commit → Push
  • Create PR + ask someone to review

In short: pull, branch, work, commit, push, PR, review, merge. Follow the circle and the whole group works in the same project — without anyone overwriting anyone else.
