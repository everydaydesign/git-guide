Så fungerar Git

Git sparar ögonblicksbilder av er kod, så att flera personer kan arbeta i samma projekt utan att skriva över varandra. Här är hela bilden, från din första commit till en mergad Pull Request.

01
Grunden

En tidsmaskin för er kod

I stället för att spara kopior som v1, v2_final, FINAL håller Git koll på varje ändring åt dig.
Du kan alltid gå tillbaka till en tidigare version — och se vem som ändrat vad, och när.

Den gamla vägen

projekt_v2.php

Manuella kopior. Ingen vet vilken som är senast — och två personer skriver lätt över varandra.

02
Stora bilden

Så flödar koden i en grupp

En person skapar repot och bjuder in de andra.
Alla hämtar (pull) senaste versionen, jobbar var och en på sin egen branch, och skickar tillbaka sitt arbete via en Pull Request som granskas innan den slås ihop.

Gyllene regeln: jobba aldrig direkt i main — skapa alltid en egen branch.

03
Grundläggande

Stage och commit — som att packa en väska

Först väljer du vad som ska med (stage). Sen stänger du väskan och sätter en etikett (commit). Två steg, av en anledning.

Steg 1 · Stage
Välj vad som ska med
Som att välja kläder till väskan — du plockar vilka ändrade filer som hör ihop i just den här sparningen.
login.phpstaged
style.cssstaged
test.phpej vald
Steg 2 · Commit
Stäng väskan, sätt etikett
Du sparar en ögonblicksbild av de valda filerna — med ett meddelande som beskriver vad du gjort.
COMMIT-MEDDELANDE
Lade till inloggningssida

Commit

sparar 2 filer lokalt

Varför två steg? Du kanske har ändrat fem filer men vill bara spara de två som hör ihop just nu. Med stage väljer du exakt vad som ska ingå — så att varje commit blir en logisk, begriplig ändring.

05
Branches

En egen arbetsyta, skild från main

En branch är en kopia av projektet där du kan jobba fritt utan att påverka andras kod. När du är klar slås din branch ihop med main via en Pull Request.

Gyllene regeln: jobba aldrig direkt i main. Skapa alltid en branch först.

Bra branch-namn
  • contact
  • products
  • login
  • startpage

Pull hämtar andras ändringar från GitHub till dig. Push skickar dina commits upp. Du synkar alltid åt båda hållen.

07
Granskning

Ingen kod går in i main ogranskad

En Pull Request låter gruppen läsa din kod innan den slås ihop. En annan person granskar — den enda regeln är att du inte granskar din egen kod.

Jobba i din branch

Du (författare)

Push till GitHub

din branch laddas upp

Skapa Pull Request

beskriv vad du gjort

Granskare läser

en annan gruppmedlem
Granskarens beslut
  • Approve
    Allt ser bra ut → PR:n kan mergas till main.
  • Comment
    Frågor eller förslag → diskutera, ingen blockering.
  • Request changes
    Något behöver fixas → författaren ändrar och pushar igen.
08
När det krockar

När två ändrar samma rad

En konflikt uppstår när två personer ändrat exakt samma rad. Git vet inte vilken version som ska gälla — så du får välja. Oftast är svaret att kombinera båda.

Konflikt i header.php
<<<<<<< HEAD (Erik)
<a href="products.php">Produkter</a>
=======
<a href="contact.php">Kontakt</a>
>>>>>>> login (Lisa)
Lösning · behåll båda
<nav>
  <a href="products.php">Produkter</a>
  <a href="contact.php">Kontakt</a>
</nav>
Ta bort <<<, === och >>>, spara, och committa.

Undvik konflikter: prata i gruppen om vem som jobbar med vilken fil — och tillämpa funktionaliteten pull ofta.

Snabb checklista

Gör en gång · projektstart
  • En person skapar repot (Use this template)
  • Bjud in gruppen som collaborators
  • Alla accepterar och skapar sin Codespace
  • Bestäm vem som gör vad
Gör varje arbetspass
  • Byt till main + Pull
  • Skapa / byt till din branch
  • Stage → Commit → Push
  • Skapa PR + be någon granska

Kort sagt: pull, branch, jobba, commit, push, PR, granska, merge. Följ cirkeln så jobbar hela gruppen i samma projekt — utan att någon skriver över någon annan.
